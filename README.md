# Creator Club

A small, focused service that answers the central question behind any
membership-driven creator platform (think Patreon, Substack, or YouTube
Memberships):

> **Given this fan and this post, can they see it?**

A creator publishes posts. Some are free, some are reserved for paying members
at a particular tier. A fan pays to subscribe to a creator at a given tier, and
that paid membership is what unlocks the locked content. The job of this service
is to make a single, authoritative entitlement decision for any viewer/post pair,
and to drive a creator page where content visibly **locks and unlocks** based on
who is logged in and what they pay for.

Payments are central rather than an afterthought. Stripe is the system of record
for who is subscribed, at what tier, and until when, so the access decision is
always grounded in real billing state instead of a guess the client makes. The
service integrates Stripe Checkout for upgrades and downgrades, and treats Stripe
webhooks as the only writer of the fields that govern access, which keeps money
and visibility in sync.

There is one kind of account: a **user**. Any user can act as a *creator* (by
defining tiers and publishing posts) and as a *fan* (by holding a paid membership
to another user). "Creator" and "fan" are roles a user plays, not separate records.

### Payments

Stripe is the system of record for money and membership state. The frontend never
creates charges directly — the API verifies the requested action, then redirects
the user to Stripe Checkout. Entitlement-controlling fields (who has an active
membership, at what tier, until when) are only ever written in response to a
verified Stripe webhook, so the access decision always reflects the real billing
state rather than an optimistic client guess.

### Stripe Webhooks Handling
Stripe is authoritative on who has an active membership and at what tier. The related fields that control who can see posts are only updated by the worker lambda consuming the webhook events via AWS SQS. This system is setup to handle varying amounts of load because of this. If this part goes down then no memberships can be updated.

Step by Step what happens when stripe sends a webhook event:
1) Stripe sends an event to the API Gateway which hands it off to the Lambda function.
2) The Lambda verifies the event by getting the secret from the secrets manager. Once it is verified it pushes the message onto the queue and then sends 200 back to Stripe. It is important to send a success response back to Stripe only after we have stored the message (written to disk or pushed to queue).
3) The worker Lambda then pulls messages off of the queue. Anything that errors here gets pushed back onto the queue to be retried. Any messages that have too many retries go into the Dead Letter Queue which would have an alarm set so a person can review what went wrong.
4) This Lambda checks for an existing idempotency key (generated by Stripe) and then handles the event by updating the necessary fields in the main DB. Stripe events can also come out of order, so this Lambda must handle that as well.

I would probably move the payments module out to its own service as well, so that it is isolated from the main api. This ensures that the blast radius from faults is lessened and lets this service scale independently.

# Missing Features

### Media (Photos and Videos)
I wanted to keep the project simple and just focus on the membership access and payments integration parts. Below is a high level plan for implementing this.

Media would be stored in an S3 bucket with metadata in the PostgreSQL DB. Users can upload media with a post via presigned urls. Not much changes about the posts, except what gets teased if a post is locked. It is no longer just some text.
For photos maybe send a blurred version of the photo to the client, or just the top part cropped. You could even let the user who uploaded the photo edit it to customize what the teaser looks like.

For videos, a good teaser is to allow the user to upload a section of the video (up to some length like 2min). Then, you would serve that to the client.

### Discovery
Discovery (recommendations and ranking) is its own large problem space (search relevance, social-graph signals, and cold-start for new creators) so I scoped it out to keep the focus on the entitlement model. Building it well would mean treating it as a dedicated system rather than a feature bolted onto this service.

### Post Feed
A basic post feed is just all the users' posts who you are subscribed to in a list in reverse chronological order. Adding any kind of algorithm is more complex and falls into similar categories as discovery.
